# Assignment 3 Changes + Viva Notes (CYLite)

This document explains what was changed, why it was changed, and how to discuss the concepts in viva.

## A) What changed in code

## 1. Grammar conflict resolution (Part 1 + Part 2)

File changed:
- `cylite/yapl.y`

Major change:
- Refactored statement handling using matched/unmatched strategy across all constructs that can carry statements.

Why this was required:
- Simple matched/unmatched only for `if` is not enough when other productions (labels, loops, switch) can contain nested statements, because `else` can still become ambiguous in LALR state merging.

What was done:
- Added `matched_labeled_statement` and `unmatched_labeled_statement`.
- Split switch into matched/unmatched forms.
- Split loop forms into:
  - `matched_iteration_statement`
  - `unmatched_iteration_statement`
- Extended same idea to CYLite forms that embed statements:
  - `foreach`
  - `for (id in range(l, r))`
- Added explicit `elif` support in conflict-safe form:
  - `if (...) ... elif (...) ... else ...`
  - `if (...) ... elif (...) ...` (no final else)

Extra ambiguity removed:
- Range-loop bounds changed from `expression` to `assignment_expression` in grammar to avoid comma-related reduce/reduce ambiguity against grammar rule `expression -> expression ',' assignment_expression`.

Result:
- No shift/reduce and no reduce/reduce conflicts with:

```sh
bison -y -d -Wcounterexamples yapl.y
```

## 2. Reverse derivation (Part 3)

File changed:
- `cylite/yapl.y`

What was added:
- A reduction recorder using in-memory array of strings.
- High-level rule actions call `record_derivation("A -> B ...")`.
- On parse success, reductions are printed in reverse order.

Output marker:

```text
=== Reverse Derivation (high-level reductions) ===
```

Note for viva:
- This is a practical high-level reverse derivation trace (major grammar rules), not a full concrete parse tree print for every terminal-level reduction.

## B) Concepts to explain in viva

## 1. LALR(1) automaton

Core point:
- LALR(1) merges LR(1) states with same LR(0) core to keep table small.
- Merging can introduce conflicts if lookaheads become less specific.

Talking point:
- "We used Bison-generated states and counterexamples to identify exact conflict contexts."

## 2. Shift/Reduce conflict

Definition:
- Parser can either shift next token or reduce a rule, both seemingly valid.

In this assignment context:
- `else` attachment in nested `if` contexts (dangling-else family).

Resolution strategy used:
- Structured matched/unmatched grammar and propagated it through every statement-carrying construct.

## 3. Reduce/Reduce conflict

Definition:
- Two different reductions are both possible with same lookahead.

In this assignment context:
- Comma in range bounds with `expression` collided with expression-list production.

Fix:
- Restrict range bounds to `assignment_expression`.

## 4. Reverse derivation idea

Concept:
- Bottom-up parsers reduce RHS to LHS.
- Sequence of reductions is reverse of a derivation sequence.

What we print:
- A high-level reverse reduction trace after successful parse.

## 5. ACTION vs GOTO table

ACTION table:
- On terminals, gives `shift`, `reduce`, `accept`, or error.

GOTO table:
- On non-terminals, gives next state after reduction.

Why matrix export matters:
- Easier to inspect in report/slides and aligns with lecture-style tabular format.

## 6. Error diagnostics

Old behavior:
- Generic syntax error.

New behavior:
- Includes line and nearest token, so debugging invalid source is faster.

## C) Quick viva Q/A prep

1. Why not only fix dangling-else in `if` rules?
- Because loops/switch/labels also carry statements; ambiguity can reappear through those paths in LALR merged states.

2. Why switch to `assignment_expression` in range?
- To avoid comma ambiguity with expression-list grammar and remove reduce/reduce conflict.

3. Is reverse derivation same as parse tree?
- Related, but not identical in this implementation. We print reverse high-level reductions, which is a compact derivation trace.

4. How did you verify conflict removal?
- With `bison -y -d -Wcounterexamples yapl.y`; no conflict warnings after refactor.

5. How to demonstrate parsing table output quickly?
- Run `make a3-table` and open `a3_artifacts/action_table.csv` and `a3_artifacts/goto_table.csv`.
